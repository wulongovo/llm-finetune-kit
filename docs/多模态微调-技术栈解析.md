## 1. VLM 架构总览

### 什么是 VLM（Vision-Language Model）

VLM 是能够同时理解图像和文本的多模态大模型。与纯文本LLM不同，VLM具备：

| 能力 | 纯文本 LLM | VLM |
|------|-----------|-----|
| 输入 | 仅文本 | 图像 + 文本 |
| 视觉理解 | 无 | 图像描述、目标检测、OCR |
| 跨模态推理 | 无 | 图文问答、图表分析 |
| 典型应用 | 文本生成/问答 | 文档理解、医疗影像、工业质检 |

### VLM 的通用架构

```
输入: [图片] + "这张图中有什么？"
         |              |
         v              v
+------------------+   +-----------+
|  Vision Encoder  |   | Tokenizer |
|  (ViT/SigLIP)    |   |           |
+--------+---------+   +-----+-----+
         |                    |
         v                    v
    图像特征 tokens      文本 tokens
         |                    |
         +--------+-----------+
                  |
                  v
         +--------+---------+
         |   Multimodal     |
         |   Projector      |  ← 将图像特征映射到文本特征空间
         | (MLP/Q-Former)   |
         +--------+---------+
                  |
                  v
         +--------+---------+
         |   LLM Backbone   |
         | (Qwen/InternVL)  |
         +--------+---------+
                  |
                  v
         输出: "图中有一只橘色的猫坐在沙发上。"
```

### 本项目支持的两种架构

| 架构 | Vision Encoder | LLM | Projector | 特点 |
|------|---------------|-----|-----------|------|
| Qwen2.5-VL | ViT (自研) | Qwen2.5 | MLP | 原生支持动态分辨率 |
| InternVL | InternViT-6B | InternLM2 | MLP | 动态分辨率 + 多图支持 |

### 面试要点

- VLM 和多模态 Embedding（如 CLIP）的区别？-> CLIP 只做图文对齐（检索），VLM 能生成文本回答
- Vision Encoder 为什么用 ViT 而不是 CNN？-> ViT 的全局注意力更适合理解图像整体语义
- Projector 的作用？-> 将图像特征空间映射到文本特征空间，让 LLM 能"看懂"图像

## 2. 模型工厂 (ModelFactory)

### 原理

不同 VLM 架构的加载方式、对话模板、图像处理流程各不相同。模型工厂将这些差异封装在统一接口后面。

```python
class ModelFactory:
    def __init__(self, model_name_or_path, strategy="qlora"):
        self.model_name = model_name_or_path
        self.strategy = strategy
        self.architecture = detect_model_architecture(model_name_or_path)

    def load(self, device_map="auto", torch_dtype=torch.bfloat16):
        """统一加载接口，返回 (model, tokenizer, processor)"""
        if self.architecture == "qwen2_vl":
            # Qwen2-VL: 使用专用的 Qwen2VLForConditionalGeneration
            from transformers import Qwen2VLForConditionalGeneration
            model = Qwen2VLForConditionalGeneration.from_pretrained(...)
            processor = AutoProcessor.from_pretrained(...)  # 处理图像+文本
        elif self.architecture == "internvl":
            # InternVL: 使用通用 AutoModel
            model = AutoModel.from_pretrained(...)
            tokenizer = AutoTokenizer.from_pretrained(...)
```

### 架构自动检测

通过正则匹配模型名称 + 解析 config.json 实现自动检测：

```python
_ARCH_PATTERNS = {
    "qwen2_vl": [r"qwen2[\.\-]?vl", r"qwen[\.\-]?vl"],
    "internvl": [r"internvl", r"intern[\.\-]?vl"],
}

def detect_model_architecture(model_name_or_path):
    name_lower = model_name_or_path.lower()
    for arch, patterns in _ARCH_PATTERNS.items():
        for pat in patterns:
            if re.search(pat, name_lower):
                return arch
    # fallback: 从 config.json 的 architectures 字段检测
    ...
```

### LoRA target_modules 自动选择

不同架构的注意力层命名不同，模型工厂自动选择：

```python
_LORA_TARGET_MODULES = {
    "qwen2_vl": ["q_proj", "k_proj", "v_proj", "o_proj", "gate_proj", "up_proj", "down_proj"],
    "internvl": ["qkv", "proj", "fc1", "fc2", "gate_proj", "up_proj", "down_proj"],
}
```

### 面试要点

- 为什么用工厂模式而不是 if-else？-> 工厂模式封装了创建逻辑，新增架构只需添加一个分支，不修改上层代码
- target_modules 怎么确定的？-> 通过打印模型结构 `print(model)` 找到所有 Linear 层的名字，选择注意力相关的层
- 为什么不直接用 AutoModelForCausalLM？-> Qwen2-VL 有专用的类，AutoModel 可能加载错误的架构

## 3. QLoRA / LoRA / Full 三种微调策略

### 策略对比

| 维度 | QLoRA | LoRA | Full |
|------|-------|------|------|
| 基础模型精度 | 4bit NF4 | bf16 (16bit) | bf16 (16bit) |
| 可训练参数 | LoRA 适配器 (~0.1-1%) | LoRA 适配器 (~0.1-1%) | LLM 全部参数 (~70%) |
| 显存需求 (7B) | ~8GB | ~16GB | ~30GB |
| 训练速度 | 中等 | 较快 | 较慢 |
| 效果 | 接近 LoRA | 接近 Full | 最佳 |
| 适用场景 | 消费级GPU | 追求效果 | 追求极致 |

### QLoRA 核心原理

```
原始模型 (fp16, ~14GB)
    |
    v
4bit NF4 量化 (~3.5GB)  ← BitsAndBytes 量化
    |
    v
附加 LoRA 适配器 (r=16, ~几十MB)  ← PEFT
    |
    v
训练时: 前向传播用量化权重，反向传播只更新 LoRA
    |
    v
推理时: LoRA 权重合并回原始模型
```

**NF4 量化**：基于正态分布的最优4bit量化方案，比普通 INT4 更精确
**Double Quantization**：对量化常数再量化，额外节省 0.5GB 显存
**Paged AdamW**：优化器状态分页到 CPU 内存，避免显存 OOM

### 项目中的实现

```python
# QLoRA 配置
def get_quantization_config(strategy):
    if strategy == "qlora":
        return BitsAndBytesConfig(
            load_in_4bit=True,
            bnb_4bit_quant_type="nf4",
            bnb_4bit_compute_dtype=torch.bfloat16,
            bnb_4bit_use_double_quant=True,
        )

# 应用 LoRA
def _apply_lora(self):
    lora_config = LoraConfig(
        r=self.config.lora_r,           # 16
        lora_alpha=self.config.lora_alpha,  # 32
        lora_dropout=self.config.lora_dropout,  # 0.05
        target_modules=get_lora_target_modules(self.architecture),
        task_type=TaskType.CAUSAL_LM,
    )
    self.model = get_peft_model(self.model, lora_config)
```

### 面试要点

- QLoRA 的 NF4 是什么？-> NormalFloat4，基于正态分布的最优4bit量化，比 INT4 的均方误差更小
- LoRA 的 r 和 alpha 怎么选？-> r 一般 8-32，alpha 一般 2*r。r 越大拟合能力越强但显存越多
- Double Quantization 的作用？-> 量化常数本身也占显存（约 0.5GB/7B模型），再量化后节省这部分
- 为什么 QLoRA 用 paged_adamw_8bit？-> 优化器状态（momentum + variance）约占模型大小的 2 倍，分页到 CPU 避免 OOM

## 4. 多模态数据处理管线

### 三种数据格式

```
VQA 格式: {"image": "path", "question": "...", "answer": "..."}

Caption 格式: {"image": "path", "caption": "..."}

Conversation 格式: {"image": "path", "conversations": [
    {"role": "user", "content": "<image>\n问题"},
    {"role": "assistant", "content": "回答"}
]}
```

### 统一处理流程

```
原始 JSONL 文件
    |
    v
自动检测格式 (detect_data_format)
    |
    v
统一转换为 Conversation 格式 (convert_to_conversation)
    |
    v
MultimodalDataset 封装 (图片加载 + 文本编码)
    |
    v
collate_fn 批处理 (根据架构选择不同的处理方式)
    |
    v
模型输入: {input_ids, attention_mask, pixel_values, labels}
```

### 架构特定的 collate_fn

```python
# Qwen2-VL: 使用 processor 的 chat 模板
text = processor.apply_chat_template(messages, tokenize=False)
inputs = processor(text=text, images=images, return_tensors="pt")

# InternVL: 自定义对话模板
prompt = f"<image>\nUser: {question}\nAssistant:"
inputs = tokenizer(prompt, return_tensors="pt")
```

### 面试要点

- 为什么要统一数据格式？-> 上层训练代码只需处理一种格式，新增数据源只需写一个转换函数
- `<image>` 标记的作用？-> 告诉模型图片应该插入在文本的哪个位置，是 VLM 的标准做法
- collate_fn 为什么按架构分？-> 不同模型的 processor 对图片的预处理方式不同（resize 尺寸、normalize 参数等）

## 5. Qwen2.5-VL 架构详解

### 模型结构

```
Qwen2.5-VL-7B-Instruct
├── visual (Vision Encoder)
│   ├── patch_embed: 将图片分割为 14x14 的 patch
│   ├── pos_embed: 位置编码（支持动态分辨率）
│   └── blocks: 32 层 ViT Transformer
├── model (LLM Backbone)
│   ├── embed_tokens: 文本嵌入层
│   ├── layers: 32 层 Transformer Decoder
│   │   ├── self_attn: 注意力层 (q_proj, k_proj, v_proj, o_proj)
│   │   ├── mlp: FFN 层 (gate_proj, up_proj, down_proj)
│   │   └── input_layernorm / post_attention_layernorm
│   └── norm: 最终 LayerNorm
└── lm_head: 输出投影层
```

### 关键特性

- **动态分辨率**：原生支持不同尺寸的图片输入，无需 resize 到固定尺寸
- **Naive Dynamic Resolution**：将图片按 28x28 的窗口分割，不同分辨率产生不同数量的 visual tokens
- **3D RoPE**：对视频帧使用 3D 旋转位置编码（时间+高度+宽度）

### 项目中的使用

```python
# 加载
from transformers import Qwen2VLForConditionalGeneration, AutoProcessor
model = Qwen2VLForConditionalGeneration.from_pretrained("Qwen/Qwen2.5-VL-7B-Instruct")
processor = AutoProcessor.from_pretrained("Qwen/Qwen2.5-VL-7B-Instruct")

# 推理
messages = [{"role": "user", "content": [
    {"type": "image", "image": pil_image},
    {"type": "text", "text": "描述这张图片"},
]}]
text = processor.apply_chat_template(messages, tokenize=False, add_generation_prompt=True)
inputs = processor(text=[text], images=[pil_image], return_tensors="pt")
output = model.generate(**inputs, max_new_tokens=512)
```

## 6. InternVL 架构详解

### 模型结构

```
InternVL3-8B
├── vision_model (InternViT-6B)
│   ├── embeddings: Patch Embedding
│   ├── encoder: 48 层 ViT Transformer
│   └── layernorm: 最终 LayerNorm
├── mlp1 (Projector)
│   └── 2 层 MLP: 将视觉特征映射到文本空间
├── language_model (InternLM2)
│   ├── model: 32 层 Transformer Decoder
│   └── lm_head: 输出投影
└── img2tok: 动态分辨率 token 映射
```

### 关键特性

- **动态分辨率**：支持从 224x224 到 4096x4096 的任意分辨率
- **多图支持**：原生支持多张图片输入
- **Progressive Training**：分阶段训练，先低分辨率后高分辨率

### 项目中的使用

```python
# InternVL 使用 AutoModel 加载
from transformers import AutoModel, AutoTokenizer
model = AutoModel.from_pretrained("OpenGVLab/InternVL3-8B", trust_remote_code=True)
tokenizer = AutoTokenizer.from_pretrained("OpenGVLab/InternVL3-8B", trust_remote_code=True)

# InternVL 内置 chat 方法
response, history = model.chat(tokenizer, pil_image, "描述这张图片", 
                                generation_config={"max_new_tokens": 512})
```

## 7. Vision Encoder 冻结策略

### 为什么要冻结

```
全量微调参数分布:
  Vision Encoder: ~600M 参数 (预训练好的，不需要改)
  LLM Backbone:   ~7B 参数   (需要微调以理解视觉特征)
  Projector:      ~100M 参数 (需要微调以对齐特征空间)

如果全部训练:
  1. 显存翻倍（需要存储梯度和优化器状态）
  2. 容易过拟合（微调数据量远小于预训练数据）
  3. 视觉理解能力退化（破坏预训练的视觉特征）
```

### 项目中的实现

```python
def _setup_full_finetune(self):
    """全量微调：冻结 Vision Encoder，只训练 LLM"""
    for name, param in self.model.named_parameters():
        if "visual" in name or "vision_model" in name:
            param.requires_grad = False
```

### 面试要点

- 冻结多少参数合适？-> 通常冻结 Vision Encoder（60-70%参数），只训练 LLM + Projector
- 什么时候不冻结？-> 当微调数据量很大（>100K）且任务与预训练差异大时（如医学影像）
- LoRA 也需要冻结 Vision Encoder 吗？-> 不需要，LoRA 只作用于 LLM 的 Linear 层，Vision Encoder 本身就不受影响

## 8. 显存优化技术

### 显存占用分析

```
7B VLM 模型显存分布 (QLoRA):
  模型参数 (4bit):    ~3.5 GB
  LoRA 参数 (bf16):   ~0.05 GB
  优化器状态:          ~0.1 GB  (paged to CPU)
  激活值:             ~2-4 GB  (gradient checkpointing 后)
  图像特征:           ~0.5-1 GB
  总计:               ~6-9 GB
```

### 优化技术一览

| 技术 | 节省显存 | 原理 |
|------|---------|------|
| 4bit QLoRA | ~75% | NF4 量化模型参数 |
| Gradient Checkpointing | ~30-50% | 前向时不保存激活值，反向时重新计算 |
| Paged AdamW | ~2GB | 优化器状态分页到 CPU |
| DeepSpeed ZeRO-2 | ~30% | 分片优化器状态和梯度 |
| bf16 混合精度 | ~50% (vs fp32) | 使用 16bit 训练 |

### Gradient Checkpointing 原理

```
正常训练:
  Layer1 → 保存激活 → Layer2 → 保存激活 → Layer3
  反向: 直接用保存的激活计算梯度

Gradient Checkpointing:
  Layer1 → 不保存 → Layer2 → 不保存 → Layer3
  反向: 重新计算 Layer1/2 的激活，再计算梯度
  代价: 多 30% 计算时间，省 50% 显存
```

## 9. DeepSpeed 分布式训练

### ZeRO-2 vs ZeRO-3

| 特性 | ZeRO-2 | ZeRO-3 |
|------|--------|--------|
| 分片内容 | 优化器状态 + 梯度 | 优化器状态 + 梯度 + 模型参数 |
| 通信开销 | 较低 | 较高 |
| 显存节省 | 中等 | 最大 |
| 适用场景 | 7B-14B 模型 | 14B+ 大模型 |

### VLM 多卡训练命令

```bash
# 2卡 ZeRO-2 训练 Qwen2.5-VL
deepspeed --num_gpus 2 train_vlm.py \
  --config configs/qwen25_vl_lora.yaml \
  --deepspeed configs/deepspeed_zero2.json
```

### 面试要点

- ZeRO 的三个阶段分别做什么？-> ZeRO-1 分片优化器状态，ZeRO-2 加上分片梯度，ZeRO-3 再加上分片模型参数
- 为什么 VLM 更需要分布式训练？-> VLM 比纯文本 LLM 多了 Vision Encoder 和图像处理的显存开销
- 通信瓶颈怎么解决？-> NVLink 互联 + overlap_comm 通信重叠

## 10. VLM 评估体系

### 四维评估框架

| 维度 | 指标 | 评估方法 | 适用任务 |
|------|------|---------|---------|
| VQA 准确率 | accuracy | 模糊匹配+关键词+数字 | 视觉问答 |
| 描述质量 | BLEU-1 + 关键词覆盖率 | n-gram 匹配 | 图像描述 |
| OCR 准确率 | 字符准确率 + 完全匹配率 | 编辑距离 | 文字识别 |
| 推理速度 | tokens/s | 计时 benchmark | 部署评估 |

### VQA 匹配策略

```python
def _match_answer(self, prediction, ground_truth):
    # 1. 精确匹配
    if pred == gt: return True
    # 2. 包含匹配（标准答案在预测中）
    if gt in pred: return True
    # 3. 数字匹配（答案中的数字都出现在预测中）
    pred_nums = set(re.findall(r'\d+\.?\d*', pred))
    gt_nums = set(re.findall(r'\d+\.?\d*', gt))
    if gt_nums and gt_nums.issubset(pred_nums): return True
    return False
```

### 面试要点

- 为什么不用 ROUGE/BLEU 评估 VQA？-> VQA 是事实性回答，关键是"对不对"而不是"像不像"
- 模糊匹配会不会误判？-> 会，所以用了三重匹配策略，数字匹配特别适合数学题
- 评估数据量多少合适？-> 至少 100 条，推荐 500+ 条以获得稳定的指标

## 12. 面试常见问题

### Q: VLM 微调和纯文本 LLM 微调有什么区别？

**A:** 三个核心区别：
1. **输入不同**：VLM 需要处理图像+文本，涉及 Vision Encoder、图像预处理、多模态 Projector 等额外组件
2. **数据格式不同**：VLM 数据需要图片路径、问题、回答三元组，且不同架构的对话模板不同
3. **显存开销更大**：图像经过 Vision Encoder 后会生成大量 visual tokens（一张 448x448 的图约 256 个 token），显著增加序列长度和显存占用

### Q: 为什么同时支持两个模型架构？

**A:** 面试时能展示"抽象和封装"的能力——通过模型工厂将不同架构的差异封装起来，上层代码不需要感知底层是哪个模型。这在实际工作中很重要，因为业务需求经常变化，今天用 Qwen-VL，明天可能就要换成 InternVL，如果每次都改训练代码就太低效了。

### Q: 微调数据量多少合适？

**A:** 取决于任务复杂度：
- 简单 VQA（是/否、计数）：500-1000 条即可
- 复杂推理（图表分析、OCR）：2000-5000 条
- 通用能力提升：10K-100K 条
关键原则：数据质量 > 数量。1000 条高质量多样化的数据 > 10000 条低质量重复数据。

### Q: 如何判断微调是否过拟合？

**A:** 三个信号：
1. 训练 loss 持续下降但 eval loss 开始上升
2. 模型在训练集上回答很好但泛化能力下降
3. 模型开始"背答案"而不是"理解问题"
解决方法：减少 epoch 数（3 通常足够）、增大 dropout、使用更多 eval 数据

### Q: QLoRA 微调后如何部署？

**A:** 两步：
1. 合并 LoRA 适配器到基础模型：`model.merge_and_unload()`
2. 用 vLLM 加载合并后的模型，支持 OpenAI 兼容 API
vLLM 的 PagedAttention 对 VLM 特别友好，因为图像 token 数量不固定，PagedAttention 可以动态分配显存。
